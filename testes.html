01 · Contexto

O que é a Ryzon API.

API REST que recebe denúncias ambientais (queimadas, garimpo ilegal e alagamentos), valida coordenadas contra fontes oficiais, analisa imagens com IA (Gemini) e escala automaticamente para IBAMA, MPF e Defesa Civil quando não há resposta em 72h.

STACK

NestJS 11 + Prisma

TypeScript estrito, PostgreSQL, Jest como test runner. pnpm como package manager.

ARQUITETURA

DDD em 3 camadas

Domínio puro isolado de infraestrutura. Regra de negócio sem amarra com framework.

QUALIDADE

14 specs · TDD

Cada caso de uso nasce do seu teste. Suíte completa roda em menos de 2 segundos.

02 · DDD & Clean Architecture

O núcleo não conhece o mundo.

No DDD, entidades e regras de negócio ficam no centro. Quanto mais externa a camada, mais técnica e substituível ela é.

Onion Architecture · entidades no centro, infraestrutura na borda

Por que esse formato?

A regra de ouro: dependência aponta pra dentro. O domínio não importa Prisma, nem Express, nem NestJS — quem se adapta é a infraestrutura.

  • Entidades — modelo de negócio puro
  • Use Cases — orquestram regras
  • Controllers / Presenters — traduzem para HTTP
  • Infrastructure — DB, libs, rede
03 · No projeto ryzon-api

Três pastas. Uma direção.

Cada pasta de src/ corresponde a um anel do diagrama anterior. O domínio fica no centro.

Domínio src/domain/
Entidades (DenunciaEntity, StatusHistoryEntity), contratos (abstract repositories, ports como GeoValidationPort e ImageAnalysisPort) e use-cases (CreateDenunciaUseCase, AnalyzeImageUseCase…). TypeScript puro — zero @Injectable(), zero import de Prisma.
testável sem mock
Infraestrutura src/infrastructure/
Adapters que implementam as portas: PrismaDenunciasRepository, InMemoryDenunciasRepository, GeminiService, TelegramService. Trocar Prisma por Drizzle só mexe aqui — o domínio não muda uma linha.
substituível
Módulos / HTTP src/modules/
Controllers, DTOs com class-validator, guards JWT e factories que injetam as implementações concretas nos use-cases. Camada fina — só traduz HTTP em chamadas de domínio.
borda externa
04 · Setas representam dependência

Quem chama quem.

No fluxo real do projeto, Controllers chamam Use Cases que dependem de interfaces — Infrastructure só entra via injeção de dependência.

Frameworks & Drivers → Adapters → Application → Entities

O que esse desenho prova

As setas nunca apontam pra fora do círculo. Infrastructure implementa interfaces que vivem em Application — esse é o Dependency Inversion Principle (DIP) em ação.

  • Mudar o ORM = mexer só em Infrastructure
  • Mudar HTTP por gRPC = mexer só em Frameworks
  • Mudar regra de negócio = mexer só em Entities
05 · Inversão de dependência na prática

O mesmo use-case não sabe
quem está executando ele.

A interface (porta) vive no domínio. As implementações (adapters) vivem fora. Em teste, injetamos in-memory. Em produção, injetamos Prisma. O use-case é idêntico.

Spec / Teste
create-denuncia.spec.ts
Use-Case + Porta
CreateDenunciaUseCase
Adapter
InMemory · Prisma
Porta = abstract class no domínio. Define o contrato (assinaturas de métodos) sem implementar nada.
Adapter = classe concreta na infraestrutura. Implementa a porta com a tecnologia escolhida (Prisma, in-memory, HTTP).
Factory no module escolhe qual adapter injetar em runtime — abrindo espaço para feature flags, A/B e ambientes diferentes.
06 · TDD

Red. Green. Refactor.

TDD não é só "escrever teste antes". É um design tool: forçar você a usar a API antes de implementá-la faz emergir contratos mais limpos e dependências mais finas.

passo 1
RED · escreve o teste
Descreve o comportamento esperado com it('deve…'). O teste falha — o use-case ainda nem foi criado. Esse erro é a especificação executável.
passo 2
GREEN · faz passar
Implementa o mínimo para o teste passar. Sem otimização, sem generalização precoce. Verde é a meta — funcionalidade comprovada.
passo 3
REFACTOR · melhora
Limpa duplicação, extrai métodos, renomeia. A bateria de testes vira sua rede de segurança — refatorar deixa de ser ato de fé.
07 · Estratégia de testes

Por que in-memory repository
e não mocks de Prisma.

Mockar prisma.denuncia.findMany testa como o código chama Prisma — não se a regra de negócio funciona. Implementar um repositório real, em memória, testa o comportamento.

Mock de Prisma

  • Acopla teste à API do Prisma — muda o ORM e tudo quebra.
  • Verifica chamadas (toHaveBeenCalledWith), não resultado.
  • Pode passar verde com bug — você programou o mock pra retornar o que esperava.
  • Fragilidade: refactor benigno faz teste falhar.

InMemoryRepository

  • Implementa a mesma interface que o repositório real.
  • Estado real: create de fato adiciona ao items[].
  • Testes verificam o que o sistema faz, não como faz.
  • Refactor livre — só quebra quando o comportamento muda.

Taxonomia de Test Doubles (Gerard Meszaros) — o que usamos aqui é um Fake:

Dummy
Preenche parâmetro, nunca é usado.
Stub
Devolve respostas prontas.
Fake
Implementação simples mas funcional.
Spy
Grava chamadas que recebeu.
Mock
Verifica expectativas pré-programadas.
08 · Anatomia de um spec

Padrão Arrange · Act · Assert

Todo teste segue três blocos. Cada um responde uma pergunta diferente.

src/domain/denuncias/use-cases/create-denuncia.use-case.spec.ts
JEST · TDD
describe('CreateDenunciaUseCase', () => {
  let repository: InMemoryDenunciasRepository
  let sut: CreateDenunciaUseCase            // System Under Test

  beforeEach(() => {                              // 🅰 ARRANGE — monta o cenário
    repository = new InMemoryDenunciasRepository()
    sut = new CreateDenunciaUseCase(repository, makeGeoValidation(['QUEIMADA']))
  })

  it('deve validar denúncia quando geo retorna flags', async () => {
    const { denuncia } = await sut.execute({       // 🅱 ACT — exercita o SUT
      type: 'QUEIMADA', channel: 'TELEGRAM',
      latitude: -3.119, longitude: -60.021,
    })

    expect(denuncia.status).toBe('VALIDADA')        // 🅲 ASSERT — verifica
    expect(denuncia.flags).toContain('QUEIMADA')
    expect(denuncia.validatedAt).toBeInstanceOf(Date)
  })
})

10 · Pirâmide de testes

Muitos rápidos. Poucos lentos.

A base — testes unitários — é grande e barata. O topo (E2E) é pequeno e caro. Inverter essa pirâmide é o anti-padrão clássico.

A pirâmide clássica · unidade na base · E2E no topo

No ryzon-api

  • Unidade · 14 specs — use-cases com in-memory repos. Milissegundos.
  • Integração — combina use-cases (criar + analisar + escalar)
  • E2E — sobe Nest + Postgres + HTTP real. Só pro happy path crítico.

F.I.R.S.T — Fast · Independent · Repeatable · Self-validating · Timely

13 · supertest no E2E

Como o teste E2E faz request.

O @nestjs/testing sobe um app de verdade em memória. O supertest envia HTTP pra ele e valida a resposta — sem precisar de servidor ouvindo em porta.

test/app.e2e-spec.ts
E2E
import request from 'supertest'
import { Test } from '@nestjs/testing'
import { AppModule } from './../src/app.module'

describe('AppController (e2e)', () => {
  let app: INestApplication

  beforeEach(async () => {
    const mod = await Test.createTestingModule({ imports: [AppModule] }).compile()
    app = mod.createNestApplication()
    await app.init()
  })

  it('/ (GET)', () => {
    return request(app.getHttpServer())   // ← supertest manda HTTP no app rodando em memória
      .get('/')
      .expect(200)
      .expect('Hello World!')
  })
})
